# 笔记存储方案说明

## 当前实现（已上线）

| 层级 | 技术 | 用途 |
|---|---|---|
| 内置笔记 | 代码内数据 `src/data/notes.ts` | 站点默认展示的公开笔记 |
| 作者新增笔记 | **IndexedDB**（优先）+ localStorage 镜像 | 登录后在浏览器本机写入 |
| 会话 | `sessionStorage` | 登录态约 12 小时 |
| 导出 | Markdown 文件下载 | 便于提交 Git 做永久发布 |

**为何现在不用云端数据库？**

站点部署在 **GitHub Pages 静态托管**，没有可写后端。浏览器无法直接安全地往服务器写文件；要做「全网可见、多设备同步」的新增笔记，必须增加一层可写服务。

## 推荐演进路径

### 阶段 A（当前）— 本机 + 导出
- 适合：个人写作草稿、本机预览
- 优点：零运维、即时可用
- 缺点：换设备/清缓存会丢；访客看不到你本机新写的笔记

### 阶段 B（推荐下一步）— 轻量后端 + 数据库
可选方案：

| 方案 | 数据库 | 适合 |
|---|---|---|
| **Cloudflare D1 + Worker** | SQLite | 与静态站同生态，免费额度够个人用 |
| **Supabase** | Postgres | 自带 Auth + API，上手快 |
| **Firebase Firestore** | 文档库 | 快速原型 |
| **自建 FastAPI + SQLite/Postgres** | 任意 | 已有服务器（如 1.14.106.17）时 |

建议表结构（示意）：

```sql
notes (
  id, slug, title, summary, body_md,
  category, tags_json, status,
  author_id, created_at, updated_at, published
)
```

登录可升级为：
- 密码哈希存在服务端（bcrypt/argon2）
- 或 GitHub OAuth / Magic Link

### 阶段 C — Git-backed CMS
用 Decap / Sveltia CMS，登录后把 Markdown 提交到仓库，再自动部署。适合「笔记即代码」工作流。

## 结论

- **短期**：IndexedDB 本机存储 + 导出 MD（已实现）
- **中期若要「直接发布给所有访客」**：上数据库（优先 Cloudflare D1 或现有服务器上的 SQLite/Postgres）
- **不必一开始就上重型数据库**；等你确认写作频率与多设备需求后再接阶段 B 即可

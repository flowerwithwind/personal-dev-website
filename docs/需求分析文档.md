## 1. 项目背景与目标

### 1.1 背景

个人开发者在求职、开源协作与技术影响力建设中，需要一个**可控、可叙事、可转化**的在线身份中枢。GitHub / 博客 / 简历 PDF 各有局限：GitHub 偏仓库列表、博客偏内容、PDF 不可交互。成熟个人站将「人设 + 能力 + 作品证据 + 联系转化」集中到同一体验中。

当前工作区已具备可展示的工程资产，可作为作品集核心：

| 项目 | 定位 | 技术关键词 |
|---|---|---|
| **Code Review Agent** | LLM + Agent + RAG 智能代码评审 | FastAPI、GitHub Webhook、ChromaDB、DeepSeek |
| **智能问数 (Wenshu)** | NL2SQL + RAG + ReAct 数据问答 | Vue3、LangChain、Chart.js、BGE Embedding |
| **XHAgentOS** | 智能瞭望与智能问数 Agent 平台 | 全栈、工作流、知识库、多模块系统 |

### 1.2 产品目标

| 优先级 | 目标 | 成功标准 |
|---|---|---|
| P0 | **个人表达中枢** | 站点首先是「我的主页」：近况、笔记、项目可一眼理解 |
| P0 | **笔记 / 数字花园** | 笔记为一级导航；支持长文与可更新札记，而非仅外链博客 |
| P0 | 作品叙事 | 3 个核心项目具备问题—方案—技术—结果闭环 |
| P1 | 求职顺带覆盖 | 关于页 + 项目证据足以支撑投递，但文案不做成「纯投递站」 |
| P1 | 个人品牌 | 视觉偏现代中文个人/产品站（参考 Coze 的暖底、宋体标题、圆角） |
| P1 | 可维护 | Markdown / MDX 内容驱动，写作成本低 |
| P2 | 订阅与沉淀 | RSS、SEO / OG，支持长期更新 |

### 1.3 非目标（本期不做）

- 复杂评论社区 / 账号体系
- 付费商城 / 预约排期
- 复杂 3D 交互主站（彩蛋可选）
- 多语言完整站（可预留 i18n，首版中文为主）

### 1.4 定位修正（2026-07-16 反馈）

用户明确：

1. **风格更偏个人网站**，字体与气质参考 [Coze 官网](https://www.coze.cn/overview)（暖色纸感背景、思源宋体大标题、胶囊按钮、克制留白）。
2. **不单为求职服务**：除开源/项目外，必须包含**个人笔记**（digital garden / 公开写作）。
3. 先前 A–D 方向中「纯作品集 / 终端极客」权重过高；新增并推荐 **方向 E · Personal Soft**。

## 3. 用户与场景

### 3.1 用户画像

| 角色 | 诉求 | 行为路径 | 关键指标 |
|---|---|---|---|
| **招聘 HR / 猎头** | 快速判断匹配度 | 扫 Hero → 简历 PDF → 项目名 | 简历点击、停留时长 |
| **技术面试官** | 验证工程深度 | 项目详情 → GitHub → 技术栈 | 项目页跳出率、外链点击 |
| **潜在合作方** | 能否交付 | 服务边界 → 案例结果 → 联系 | 表单提交率 |
| **同行开发者** | 学习/Star/交流 | 项目实现细节 → Blog | GitHub stars、分享 |

### 3.2 核心用户旅程

**旅程 A · 求职初筛（最重要）**

1. 从简历链接 / 邮件签名进入首页  
2. 3 秒内读到角色定位（如「AI Agent / 全栈工程师」）  
3. 点击「查看作品」或滚动到 Projects  
4. 打开 1 个核心项目详情，确认问题与技术深度  
5. 下载简历或发邮件  

**旅程 B · 面试前突击**

1. 面试官收藏链接二次进入  
2. 直达项目详情 / 架构说明  
3. 打开 GitHub 或 Demo  

**旅程 C · 合作咨询**

1. Hero 看到可合作方向  
2. Contact 填写意图  
3. 邮件收到结构化线索  

### 3.3 使用场景与约束

| 场景 | 约束 | 设计响应 |
|---|---|---|
| 手机端 HR 碎片浏览 | 小屏、弱网 | 移动优先导航、压缩图、关键 CTA 固定可达 |
| 桌面深度阅读 | 多栏目对照 | 宽屏双栏/粘性导航 |
| 分享到微信/飞书 | 需预览图 | OG 图片与标题规范 |
| 无障碍与键盘访问 | 合规与专业感 | 焦点态、语义化、对比度 |

### 4.2 内容需求

| 内容块 | 最低完备度 | 来源建议 |
|---|---|---|
| 个人定位文案 | 1 句定位 + 3 条能力点 | 结合目标岗位（如 Code AI / 全栈） |
| 项目案例 ×3 | 问题、职责、技术、结果、链接 | 现有 README 提炼 |
| 技能列表 | 分 4 组，每组 4–8 项 | 与项目一致，避免注水 |
| 简历 PDF | 最新一版 | 与站点信息同步 |
| 头像/视觉 | 可选真实头像或抽象标识 | 统一品牌色 |
| 项目封面 | 每项目 1 张 UI 截图或架构图 | 优先真实界面 |

### 4.3 非功能需求

| 类别 | 指标 |
|---|---|
| 性能 | Lighthouse Performance ≥ 90（桌面）；首屏 LCP < 2.5s |
| 可访问性 | 对比度 WCAG AA；键盘可达；图片 alt |
| 兼容性 | 近两年版 Chrome / Edge / Safari / Firefox；iOS/Android 主流浏览器 |
| 安全 | 外链 `rel="noopener noreferrer"`；表单防 spam（honeypot/时间戳） |
| 可维护 | 组件化；内容与样式分离；中文文案可集中配置 |
| SEO | 静态可预渲染；结构化数据 Person / WebSite（可选） |

### 4.4 技术选型建议（实现阶段参考）

| 层 | 推荐 | 备选 |
|---|---|---|
| 框架 | Next.js / Astro（静态优先） | Vite + Vue/React SPA |
| 样式 | Tailwind CSS + CSS Variables | UnoCSS |
| 动效 | CSS + 少量 Framer Motion / Motion One | GSAP（过重则慎用） |
| 部署 | Vercel / Cloudflare Pages / GitHub Pages | 自建 Nginx |
| 表单 | Formspree / Getform / 自建 API | mailto only（MVP 可接受） |
| 分析 | Umami / Plausible | 无 |

> 本期交付为 **需求分析 + 高保真 HTML 原型**，不锁定最终技术栈；原型使用纯 HTML/CSS/JS 便于评审。

## 6. 页面级需求说明

### 6.1 首页 Home

**目标**：建立信任 + 引导至作品与联系。

| 区块 | 内容要素 | 交互 |
|---|---|---|
| Header | Logo、导航、简历 | 粘性；滚动后背景实色 |
| Hero | 问候、姓名、角色标签、一句话、主 CTA「查看项目」、副 CTA「联系我」 | 轻量背景装饰（网格/光晕） |
| Featured Projects | 3 张卡片 | hover 抬升；整卡可点 |
| Highlights | 3 条能力主张（Agent / 全栈 / 工程化） | 图标 + 短文 |
| About teaser | 2–3 句 + 「了解更多」 | 链到 About |
| Contact strip | 一句话邀请 + 按钮 | 链到 Contact |
| Footer | 标准页脚 | — |

### 6.2 项目列表 Projects

**目标**：浏览与筛选作品证据。

- 页头标题 + 简短说明  
- 卡片网格（2 列桌面 / 1 列手机）  
- 每卡：封面、标题、一句话、标签、年份  
- Should：标签筛选芯片  

### 6.3 项目详情 Project Detail

**目标**：支撑技术面试深聊。

建议固定章节顺序（便于简历口述对齐）：

1. **Hero**：项目名、一句话、角色、时间、状态标签  
2. **问题背景**：解决什么、谁用、为何难  
3. **方案概述**：架构图位 + 关键流程  
4. **我的贡献**：职责条目（STAR 可映射）  
5. **技术栈**：分组标签  
6. **成果**：可量化优先（准确率、耗时、模块数等；无数据则写能力证明）  
7. **外链区**：GitHub / Demo / 文档  

### 6.4 关于 About

**目标**：人设与可信履历。

- 照片/抽象标识 + 长简介  
- Timeline（教育、项目、关键节点）  
- Skills 四象限：语言 · 后端/AI · 前端 · 工程工具  
- 可选 Fun facts 降低距离感  

### 6.5 联系 Contact

**目标**：低摩擦转化。

- 说明可聊方向：实习/全职、技术交流、开源协作  
- 表单字段：姓名、邮箱、主题、留言  
- 备选：直接邮箱、GitHub、LinkedIn（如有）  
- 成功/失败反馈态

## 7. 交互与体验原则

1. **清晰优先于炫技**：动效服务层次，不挡阅读。  
2. **证据优先**：项目页信息密度高于装饰。  
3. **一次主行动**：每屏一个主 CTA（查看项目 / 下载简历 / 发邮件）。  
4. **可预期导航**：不隐藏联系方式与简历。  
5. **尊重系统偏好**：`prefers-color-scheme` / `prefers-reduced-motion`。  
6. **移动可达**：汉堡菜单；CTA 不依赖 hover。

| `--accent-dim` | `rgba(100,255,218,0.12)` | 强调浅底 |
| `--warning` | `#FFB454` | 提示 |
| `--danger` | `#FF6B7A` | 错误 |

浅色主题（Should）：背景 `#F7F9FC`，文字 `#0F172A`，强调 `#0D9488`。

### 8.3 字体（修订：对齐 Coze / 个人站阅读感）

| 用途 | 字体栈 | 说明 |
|---|---|---|
| **大标题** | `"Noto Serif SC", "Source Han Serif SC", "Songti SC", serif` | 对齐 Coze 的 SourceHanSerifCN 宋体标题气质 |
| **正文 / UI** | `"Inter", "Noto Sans SC", "PingFang SC", "Microsoft YaHei", sans-serif` | 现代无衬线，中英文混排 |
| 等宽（代码/标签可选） | `"JetBrains Mono", "Fira Code", ui-monospace, monospace` | 仅代码块使用，避免全站等宽 |

字重建议：标题 600；正文 400–500；按钮/导航 600–700。标题字号桌面约 48–58px（与 Coze 采样接近）。
